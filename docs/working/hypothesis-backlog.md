## Retired

### H-05 — skill-vs-workflow invocation comparison (retired 2026-05-12)

H-05 was reframed once and retired the same day after the author expressed doubt a second time. The retirement rationale, in order of how decisive each point is:

1. **Text-invocation patterns don't distinguish skills from workflows at the user level.** Both surfaces are reached via the same kinds of prompts — descriptive task asks, slash commands, or implicit triggers. The user is not picking between "I'd like a skill" and "I'd like a workflow"; they are asking for help with a task. Whether a given help-request lands on a skill or a workflow is downstream of routing rules, not of any signal the user emitted. Counting which surface received the dispatch tells us about routing tables, not about what users find valuable.
2. **Per-item ratios depend entirely on which denominator is chosen.** Raw totals favored skills (2221 events / 1304); per-item rates favored workflows (118.5 / 105.8); a "per item that has been invoked at least once" denominator, or a "per item published in the last N days" denominator, would each pick a different winner. The reframe (factor K=1.5 over a 30d window with item-count denominator) made the test falsifiable but did nothing to make it *meaningful*: every reasonable normalization yields a different verdict, which is the signature of a question that isn't really about ratios.
3. **The underlying question — which surface delivers more value — is better answered by user feedback than by log telemetry.** H-04 was resolved by the author calling diverge/evaluate "the number one best piece of prompting I use" — one sentence of feedback that carries more signal than any rate comparison would. The same approach (ask the author which surface they reach for and why) is the right tool for the value question H-05 was trying to proxy. Telemetry is appropriate for adoption questions ("is anyone using X at all"), not for value questions where the user can just be asked.

Lesson for future hypothesis screening: when the underlying question is "is this valuable", prefer asking over measuring. Telemetry-based hypotheses earn their place when (a) value is already established and adoption is the open question, or (b) the user cannot reliably introspect on their own behavior. Neither holds here.

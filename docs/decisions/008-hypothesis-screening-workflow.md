# 008: Hypothesis Screening — Decouple Hypotheses from Implementation

**Date:** 2026-04-06
**Status:** Proposed

## Context

The self-improvement loop currently ties hypotheses to implementation: a task is proposed, a hypothesis is attached, the task is implemented in the same round, and the hypothesis is evaluated later. The round 1 "hypothesis-before-impl" experiment tested whether defining hypotheses upfront improved CONFIRMED rates — it was REFUTED (0% vs 60% for retroactive hypotheses).

But that experiment tested the wrong thing. The original idea was not "write a hypothesis, then implement immediately and check later." It was: **generate many hypotheses without implementing anything, track them against real-world evidence, and only implement features whose underlying assumptions are validated.** The value is in the work *not done* — features whose hypotheses are refuted never consume implementation effort.

There is a second, deeper problem. The self-improvement loop gathers evidence almost entirely from within itself (gate pass rates, shellcheck results, round reports). But this repo exists to improve performance on *other projects*. Skills and workflows are consumed via symlinks in external codebases (belegarth-herald, auto_code_review, humanlayer, etc.). Evidence from those projects — whether a workflow was actually used, whether it led to project success, whether it caused friction — is far more important than whether a task passed the shellcheck gate. The self-improvement loop's internal metrics are self-referential; external project outcomes are what the user actually cares about.

## The idea

A separate workflow — not embedded in self-improvement.sh — that:

1. **Generates candidate hypotheses in bulk.** Given a set of potential improvements (from divergent design, user feedback, observed friction), produce falsifiable predictions about each one's assumptions. These are *not* task descriptions — they are claims about the world that would need to be true for the improvement to have value.

2. **Tracks hypotheses without implementation.** Hypotheses enter a tracking log with no associated task. They are evaluated periodically against evidence that accumulates naturally: usage.jsonl logs, git history in external projects, user feedback, review artifacts.

3. **Prioritizes external evidence.** Evidence from external projects (skill/workflow invocations logged by the PreToolUse hook, commit patterns in projects using symlinked workflows, user reports of friction or success) carries more weight than self-improvement loop internals. A hypothesis confirmed by external usage data is stronger than one confirmed by round-report metrics.

4. **Gates implementation on hypothesis status.** A feature is only worth implementing when its underlying hypothesis is CONFIRMED or strongly supported. REFUTED hypotheses represent saved work. INCONCLUSIVE hypotheses may need better instrumentation before any implementation is attempted.

## Example

**Candidate improvement:** "Add a complexity warning to the research-plan-implement workflow when the plan exceeds 10 steps."

**Current approach (hypothesis-before-impl):** Create a task, attach hypothesis "plans with >10 steps will trigger the warning and lead to plan simplification within 3 rounds," implement it, evaluate later.

**Proposed approach (hypothesis screening):**
- Hypothesis: "Plans generated by the RPI workflow in external projects frequently exceed 10 steps, and those plans correlate with session failures or user abandonment."
- Evidence sources: usage.jsonl (RPI invocations across projects), git log in external projects (did sessions using RPI produce commits?), user feedback.
- Track for N rounds/weeks. If evidence shows plans rarely exceed 10 steps, or that long plans succeed just as often — REFUTED. Don't build the feature. If evidence confirms the correlation — now implement, with confidence the feature addresses a real problem.

## Mechanism

### Separate script: `scripts/hypothesis-screen.sh`

This script operates independently from self-improvement.sh. It:

1. **Reads a hypothesis backlog** (e.g., `docs/working/hypothesis-backlog.md`) — a table of unimplemented hypotheses with:
   - ID, description, evidence sources, creation date, status, last-checked date
2. **Gathers evidence** from external sources:
   - `~/.claude/logs/usage.jsonl` — skill/workflow invocation frequency, which projects use what, engagement depth (reads/branch)
   - Git logs in known external project directories (configurable list)
   - `docs/reviews/` artifacts — fact-check and critique outputs from real project work
   - Hypothesis-specific grep patterns or file checks defined per hypothesis
3. **Outputs an evidence summary** per hypothesis to stdout — not a verdict, but a structured report that a human or LLM can use to judge CONFIRMED/REFUTED/INCONCLUSIVE
4. **Updates the backlog** with last-checked date and evidence summary

### Hypothesis backlog format

```markdown
# Hypothesis Backlog

| ID | Hypothesis | Evidence Sources | Created | Status | Last Checked | Evidence Summary |
|----|-----------|-----------------|---------|--------|-------------|-----------------|
| H-01 | RPI workflow is used in >2 external projects per month | usage.jsonl | 2026-04-06 | TRACKING | | |
| H-02 | Users abandon sessions when plans exceed 10 steps | usage.jsonl, git log | 2026-04-06 | TRACKING | | |
| H-03 | The bug-diagnosis workflow is never used outside self-improvement | usage.jsonl | 2026-04-06 | TRACKING | | |
```

Statuses: TRACKING (active observation), CONFIRMED, REFUTED, INCONCLUSIVE, IMPLEMENTED (hypothesis confirmed and feature built).

### Evidence hierarchy

1. **External project outcomes** — Did a workflow/skill get used in a real project? Did the session produce commits? Did the user report success or friction? (Strongest signal)
2. **Usage patterns** — Invocation frequency, engagement depth, adoption across projects. (Strong signal)
3. **Review artifacts** — Fact-check findings, critique outputs from real project work. (Medium signal)
4. **Self-improvement loop metrics** — Gate pass rates, round reports, internal hypothesis outcomes. (Weakest signal — self-referential)

### Integration with feature development

When the self-improvement loop (or a human) considers a new feature:

1. Check the hypothesis backlog for relevant hypotheses
2. If a relevant hypothesis is CONFIRMED — proceed with implementation, citing the evidence
3. If REFUTED — skip the feature, log the saved work
4. If TRACKING or INCONCLUSIVE — defer implementation, possibly add better instrumentation
5. If no relevant hypothesis exists — create one and defer implementation

This creates a natural pipeline: **ideation -> hypothesis -> evidence -> implementation**, where most candidate features never reach implementation because their assumptions don't hold.

## What this is NOT

- Not a replacement for the self-improvement loop's existing hypothesis tracking (which validates *implemented* changes)
- Not automated decision-making — the evidence report informs human judgment about whether to proceed
- Not a blocking gate — a human can override and implement despite an unconfirmed hypothesis if they have other reasons

## Consequences

- **Easier:** Feature proposals are cheaper (just a hypothesis, not a full task). Bad ideas are filtered before consuming implementation effort. External evidence becomes a first-class input to development decisions.
- **Harder:** Requires patience — hypotheses need time to accumulate evidence. Requires that the usage hook and external project symlinks are maintained. Adds a new artifact (hypothesis-backlog.md) to track.
- **Key metric:** Percentage of tracked hypotheses that are REFUTED — this directly measures implementation effort saved. A high REFUTED rate means the screening is working. (This inverts the self-improvement loop's implicit assumption that higher CONFIRMED rates are better.)

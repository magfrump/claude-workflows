# User Testing (Slim)

> Quick-start checklist. Full workflow: [user-testing-workflow.md](../user-testing-workflow.md)
> Last synced: 2026-04-08

## When to use

Planning, running, or analyzing a usability test on a feature or flow.

## Checklist

- [ ] **1. Scope** — Answer four questions before anything else:
  - What are we testing? (specific feature/flow)
  - What do we want to learn? (2-4 research questions)
  - What decisions will this inform?
  - What's the riskiest assumption?

- [ ] **2. Design sessions** — Write 3-5 tasks as user goals (not UI instructions). Prepare a moderator script with intro, warm-up, tasks, post-test debrief. Recruit 3-5 participants per round.

- [ ] **3. Run sessions** — Record screen+audio. Note-taker fills observation template (task, time, completion, errors, quotes, SEQ 1-7). Debrief after each session: top 3 issues, anything new?

- [ ] **4. Analyze** — Cluster observations by theme. Rate severity per issue (Critical/Serious/Minor/Cosmetic). Stress-test findings: is there a boring explanation? Who implements the fix?

- [ ] **5. Prioritize** — Plot issues on impact × frequency. Top-right quadrant = fix first. Administer SUS questionnaire; score < 68 = below average.

- [ ] **6. Report** — Executive summary (5 sentences max), top findings with evidence and recommendations, positive findings, next steps.

## Key rules

- Never use UI labels in task wording
- Moderator stays quiet; redirect all questions back to user
- 3-5 tasks per session (more → fatigue → bad data)
- Plan for ≥2 rounds of testing

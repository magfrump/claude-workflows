# R3 Plan: Deferred Questions Harvester Fix

## Problem

The morning summary's "Deferred Evaluation Questions" section is generated by
`_summary_deferred_evaluation` in `scripts/lib/si-morning-summary.sh`. The
function filters hypothesis-log.md rows on a single condition: Outcome
column ($7) empty. It ignores the row's Window ($5) and Checked at Round
($6) columns and never compares against the round window — so a row whose
window has not yet expired is treated identically to one that is overdue,
and the function has no notion of what the current round is.

Operationally this means the section either:
- Surfaces a row prematurely (window not yet matured) — noisy.
- Or, if the user expects "ripe" filtering, fails to surface anything new
  and only the long-stale `task-description-linter` row keeps appearing.

Spec for the correct filter: a row is a deferred question iff
`Outcome` is empty AND `Round + Window <= current_round`.

## Fix

Make the harvester explicitly aware of the current round and apply the
checkpoint-window math.

1. Extend `_summary_deferred_evaluation` to accept a `current_round`
   parameter (the most recently completed round, i.e. `end_round`).
2. For each table row whose Outcome ($7) is empty, also read Round ($2)
   and Window ($5). If both are positive integers, surface the row only
   when `Round + Window <= current_round`. If either is non-numeric or
   missing, fall back to the empty-Outcome-only behaviour so legacy log
   rows still surface.
3. Pass `$end_round` from `generate_morning_summary` into the call site.
4. Apply the same gating to `_count_deferred_hypotheses` so that the
   round-claim baseline (`deferred_hypothesis_closure`) and its evaluator
   stay consistent with what the user actually sees in the summary.
5. Update `_capture_round_claim` and `_evaluate_round_claim` to pass the
   relevant round through to `_count_deferred_hypotheses`. The capture
   uses the upcoming round (`end_round + 1`); the evaluation uses the
   round being graded.
6. Leave `_project_state_open_hypotheses` unchanged — that section
   reports rows still being tracked, regardless of whether their window
   has matured, which is a different concept from "deferred question
   ready for user input."

## Scope

- File: `scripts/lib/si-morning-summary.sh`
- Plan doc: this file
- No data changes to `docs/working/hypothesis-log.md`. The single open
  row (task-description-linter, R3+W3 → ripe at R6) continues to be the
  only row matching the spec given existing data, and that is the
  intended behaviour after the fix.

# android — Gradle / Android build dependency resolution.
#
# The SDK itself (cmdline-tools, platform-tools, one platform + build-tools) is baked
# into the image at BUILD time, which is not subject to init-firewall.sh, so these
# domains are for RUNTIME dependency resolution only — what `./gradlew` reaches when it
# resolves AndroidX/Google + Maven Central artifacts and (if not cached) the Gradle
# distribution named by the wrapper.
#
# A project that needs an SDK component NOT baked into the image (a different platform
# or build-tools version) will fail loudly rather than fetch it, exactly like uv under
# UV_PYTHON_DOWNLOADS=never: that is a rebuild of the central image, not a wider
# allowlist. dl.google.com is admitted here anyway because it is also Google's Maven
# host (google() resolves to dl.google.com/dl/android/maven2), but sdkmanager writing
# to the root-owned SDK dir is blocked at the filesystem, not the firewall.

# Google Maven (google()) — AndroidX, the Android Gradle Plugin, Google libraries.
#
# ACCEPTED RISK (like the `llm` profile's, called out deliberately): the firewall
# allowlists by destination IP, not SNI/Host (init-firewall.sh resolves A records and
# matches on dst). These hostnames resolve to Google Front End IPs, which are
# SNI-routed, so admitting them implicitly trusts the whole GFE-hosted Google surface
# — including *writable* endpoints like storage.googleapis.com — as a potential
# outbound channel for a compromised session. This is the same shared-IP weakness the
# base profile already carries (GitHub's published CIDRs, PyPI behind Fastly); Google's
# frontend is just the broadest instance. Tightening it needs an SNI-filtering egress
# proxy, not a domain edit — out of scope here, accepted knowingly.
dl.google.com
maven.google.com

# Maven Central (mavenCentral()) — the bulk of JVM/Kotlin dependencies.
repo.maven.apache.org
repo1.maven.org

# Gradle distribution (the wrapper) and the plugin portal (gradlePluginPortal()).
services.gradle.org
plugins.gradle.org
